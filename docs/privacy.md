# Приватность истории чатов

Чаты приватны на уровне API: каждый запрос фильтруется по владельцу
(`WHERE user_id = ?`), у админа нет эндпоинтов чтения чужих чатов, аудит-лог
не содержит текста запросов/ответов. Но файл БД (`data/srv-ai-ui.db`) читается
любым, у кого есть шелл на сервере под пользователем приложения или root.
Три меры защиты (дополняют друг друга):

## 1. Регламент доступа к серверу (организационная мера)

- Приложение работает под отдельным пользователем (`srv-ai`), не под root.
- SSH-доступ к серверу — только у ответственных за эксплуатацию (1–2 человека),
  по ключам, с журналированием входов (`journalctl _COMM=sshd`).
- Каталог `data/` — права `700`, владелец `srv-ai`.
- Бэкапы БД шифровать при копировании за пределы сервера
  (`age`/`gpg`), либо не выносить из контура.

## 2. Шифрование БД на диске (SQLCipher)

В `.env`: `DB_KEY=<длинная случайная строка>` (`openssl rand -base64 32`).

- Требуется пакет `sqlcipher3-binary` (в requirements указан комментарием;
  для air-gap скачать wheel заранее: `pip download sqlcipher3-binary`).
- Включать на **новой** базе. Миграция существующей:
  остановить сервис → `sqlcipher data/srv-ai-ui.db` →
  `ATTACH DATABASE 'new.db' AS enc KEY '<ключ>'; SELECT sqlcipher_export('enc'); DETACH enc;`
  → заменить файл, задать `DB_KEY`, запустить.
- Защищает: кражу диска/бэкапа, чтение файла в обход приложения.
- НЕ защищает от root на работающем сервере (ключ в `.env`). Права на `.env` — `600`.
- Потеря ключа = потеря всех данных. Ключ хранить в сейфе паролей отдела.
- Если `DB_KEY` задан, а `sqlcipher3` не установлен — приложение не стартует
  (намеренно: тихая деградация до нешифрованной БД недопустима).

## 3. Автоочистка старой истории

В `.env`: `CHAT_RETENTION_DAYS=90` (пример). Сообщения старше N дней удаляются
при старте приложения и далее раз в сутки; чаты, оставшиеся пустыми, удаляются.
`0` — хранить вечно. Заметки, события и аудит-лог не затрагиваются.
