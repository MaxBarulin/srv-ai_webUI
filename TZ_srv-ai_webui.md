# ТЗ: Корпоративный веб-интерфейс для srv-ai (АО «Адмиралтейские верфи», ОСК)

## 1. Контекст и цель

На сервере srv-ai (AMD EPYC 7763, 196 GB RAM, Ubuntu, без GPU) работает llama.cpp с моделью Qwen3.6-35B-A3B (OpenAI-совместимый API). Штатный web UI llama.cpp признан недостаточным с точки зрения ИБ (нет аутентификации, нет разграничения пользователей, нет журналирования) и будет отключён (`--no-webui`).

**Задача:** разработать собственный веб-интерфейс — единое рабочее место для 5–10 сотрудников отдела: чат с LLM, заметки, календарь, поиск по базе знаний (LightRAG). LLM должна уметь полноценно работать с заметками и календарём пользователя через tool calling по запросу на естественном языке («создай заметку…», «что у меня на этой неделе», «перенеси совещание на четверг»).

**Ключевые ограничения:**
- Сервер сейчас имеет доступ в интернет, но проект должен разворачиваться и в полностью закрытом контуре (air-gap). Никаких CDN, внешних шрифтов, внешних API в рантайме. Все зависимости — локально.
- Минимум зависимостей: весь код должен быть обозримым для аудита ИБ (цель — до ~5 тыс. строк своего кода).
- Разворачивать будет сисадмин с sudo по простой инструкции. Автор проекта админ-прав на сервере не имеет.

## 2. Стек и архитектура

- **Backend:** Python 3.10+, FastAPI + uvicorn. Один процесс, один сервис.
- **Frontend:** статические HTML/CSS/JS (vanilla ES-модули), **без этапа сборки** (никаких npm/webpack/vite). FastAPI раздаёт статику. Одностраничное приложение с тремя разделами: Чат / Заметки / Календарь.
- **БД:** SQLite (stdlib `sqlite3` или `aiosqlite`), файл в каталоге данных, WAL-режим.
- **HTTP-клиент к LLM:** `httpx` (стриминг SSE).
- **Пароли:** `bcrypt`.
- Допустимые зависимости: `fastapi`, `uvicorn`, `httpx`, `bcrypt`, `python-multipart`, `aiosqlite`, `pydantic` (идёт с FastAPI). Больше — только с обоснованием.

```
[Браузер] --HTTPS/HTTP--> [FastAPI :8080]
                             |-- /api/chat  --SSE--> llama.cpp :8000 (/v1/chat/completions)
                             |-- /api/rag   -------> LightRAG (URL из конфига, сейчас внешняя машина)
                             |-- SQLite (users, sessions, chats, messages, notes, events, audit_log)
                             |-- статика (/static, index.html)
```

Порт по умолчанию **8080**, задаётся в конфиге.

## 3. Аутентификация и роли

- Локальные логин/пароль. Самостоятельной регистрации нет.
- Роли: `admin` и `user`. Админ через раздел «Администрирование» создаёт/блокирует пользователей и сбрасывает пароли. Первый админ создаётся CLI-командой (`python -m app.create_admin`).
- Сессии: случайный токен в httpOnly + SameSite=Lax cookie, хранение сессий в БД, TTL 12 часов, продление при активности. Logout удаляет сессию.
- Пароли: bcrypt, требование к длине от 10 символов. Смена пароля пользователем в профиле.
- Rate limit на /api/login (например, 5 попыток / 5 минут на IP) — простая внутренняя реализация, без внешних библиотек.
- Все /api/* кроме login требуют валидной сессии. Статика логина доступна без сессии.

## 4. Чат с LLM

- Проксирование в llama.cpp OpenAI API (`LLM_BASE_URL`, по умолчанию `http://127.0.0.1:8000/v1`). Стриминг ответа в браузер через SSE.
- Список чатов пользователя (создать, переименовать, удалить), история сообщений в БД. Чаты приватны для пользователя.
- Модель работает в thinking-режиме (reasoning-format deepseek): reasoning-контент отображать в сворачиваемом блоке «Размышления» (по умолчанию свёрнут), в историю следующего запроса reasoning не включать.
- Рендер markdown в ответах (мини-рендерер свой или одна маленькая локальная либа, положенная в /static/vendor): заголовки, списки, таблицы, `код` с кнопкой «Копировать».
- Кнопка «Остановить генерацию» (разрыв стрима).
- Системный промпт: базовый шаблон в конфиге + автоматически подставляемые текущая дата/время и имя пользователя.
- Переключатель контекста в шапке чата: «Заметки/Календарь» (вкл. по умолчанию — даёт модели инструменты, см. §7) и «База знаний» (RAG, см. §8).

## 5. Заметки

- Две области: **Личные** (видит только владелец) и **Общие** (видят и редактируют все пользователи). У заметки: заголовок, текст (markdown), тэги, created_at, updated_at, автор, область (personal/shared).
- UI: список с поиском по подстроке и тэгам, редактор с предпросмотром markdown.
- Полный CRUD через REST API — этот же API используют инструменты LLM.
- В общих заметках фиксировать «изменено: кем, когда».

## 6. Календарь

- Личные и общие события. Поля: название, описание, дата/время начала и конца (или флаг «весь день»), место, область (personal/shared), автор.
- UI: вид «Месяц» и «Список ближайших». Создание/редактирование в модальном окне. Общие события визуально помечены.
- Полный CRUD через REST API — используется и инструментами LLM.
- Часовой пояс: Europe/Moscow, хранить в ISO 8601 c TZ.

## 7. Tool calling (LLM ↔ заметки/календарь)

llama.cpp запущен с `--jinja`, Qwen поддерживает OpenAI tools. Реализовать серверный агентный цикл:

1. Запрос пользователя уходит в LLM со списком tools.
2. Если модель вернула `tool_calls` — бэкенд исполняет их **от имени текущего пользователя** (те же права, что и в UI: чужие личные данные недоступны), результат возвращается модели, цикл продолжается (лимит: 6 итераций).
3. Финальный текстовый ответ стримится пользователю. В UI между сообщениями показывать компактные плашки «🔧 создана заметка „…“» по факту выполнения инструментов.

Набор инструментов (JSON Schema, имена и параметры зафиксировать в коде):
- `notes_search(query, scope, tags)` / `notes_get(id)` / `notes_create(title, text, scope, tags)` / `notes_update(id, ...)` / `notes_delete(id)`
- `calendar_list(date_from, date_to, scope)` / `calendar_create(...)` / `calendar_update(id, ...)` / `calendar_delete(id)`
- `get_current_datetime()`

Деструктивные действия (delete, перезапись текста заметки) — по флагу конфига `TOOLS_CONFIRM_DESTRUCTIVE=true` требуют подтверждения в UI (модель получает ответ «требуется подтверждение», пользователю показывается кнопка).

Fallback: если сервер llama.cpp не вернул корректные tool_calls, парсить JSON-блок из текста по шаблону (устойчивость к старым сборкам).

## 8. RAG (LightRAG)

- Переключатель «База знаний» в чате. При включении бэкенд перед запросом к LLM обращается к LightRAG API (`RAG_BASE_URL` из конфига; сейчас LightRAG запускается на отдельной машине разработчика, позже переедет на srv-ai — поэтому только конфиг, никакого хардкода).
- Режим запроса LightRAG (naive/local/global/hybrid) — параметр конфига, по умолчанию hybrid.
- Полученный контекст вставляется в промпт отдельным блоком с инструкцией «отвечай на основе контекста, указывай источники». Показывать в UI сворачиваемый блок «Источники».
- Эмбеддинги (bge-m3 на :8001) использует сам LightRAG — наш бэкенд к :8001 напрямую не ходит. Если LightRAG недоступен — понятная ошибка в чате, обычный режим продолжает работать.

## 9. Дизайн — фирменный стиль ОСК / Адмиралтейские верфи

Ориентир — корпоративный дайджест ОСК (приложен у заказчика) и логотип «ОСК Адмиралтейские верфи».

**Токены:**
- `--osk-red: #BE1E2D` (фирменный красный ОСК — акценты, активные элементы, кнопки-primary)
- `--osk-slate: #5B6670` (серо-стальной из логотипа — вторичный текст, иконки)
- `--osk-dark: #1E2A33` (тёмный сине-серый «штормовое море» — шапка, сайдбар)
- `--osk-paper: #FFFFFF` и `--osk-bg: #F4F5F6` (фон контента)
- `--osk-line: #D9DDE0` (разделители)

**Типографика:** заголовки — узкий гротеск в верхнем регистре с разрядкой (в духе дайджеста); использовать локально положенные свободные шрифты, например Oswald/«PT Sans Narrow» для дисплея и Inter/«PT Sans» для текста (файлы .woff2 в /static/fonts, без внешних загрузок). Кириллица обязательна.

**Сигнатурные элементы стиля** (из дайджеста): скошенные диагональные срезы блоков (clip-path на карточках и шапке), тонкие линии-«галсы» у заголовков разделов, красная вертикальная плашка у активного пункта меню — как нумерация страниц в дайджесте. Логотип предприятия в шапке слева (файл заказчик положит в /static/img/logo.svg|png).

Тёмный сайдбар (Чат / Заметки / Календарь / Администрирование), светлая рабочая область. Сдержанно: одна акцентная краска — красный, без градиентов и теней-свечений. Адаптив до ширины ноутбука (мобильная версия не требуется, но не должна разваливаться).

## 10. Схема БД (SQLite)

```
users(id, login UNIQUE, pass_hash, display_name, role, is_active, created_at)
sessions(token PK, user_id, created_at, expires_at, ip)
chats(id, user_id, title, created_at, updated_at)
messages(id, chat_id, role, content, reasoning, tool_calls_json, created_at)
notes(id, owner_id, scope, title, body, tags, created_at, updated_at, updated_by)
events(id, owner_id, scope, title, description, location, starts_at, ends_at, all_day, created_at, updated_at, updated_by)
audit_log(id, user_id, action, object_type, object_id, details, created_at, ip)
```

`audit_log` пишется на: login/logout (в т.ч. неудачные), создание/удаление пользователей, все вызовы инструментов LLM, удаление заметок/событий. Просмотр — только админу.

## 12. Развёртывание

Сценарий для сисадмина (README с точными командами):
1. Онлайн-вариант (пока есть интернет): `git clone` / копия репо → `python -m venv venv` → `pip install -r requirements.txt` → `create_admin` → systemd-юнит (файл `deploy/srv-ai-ui.service` в репо) → `systemctl enable --now`.
2. Офлайн-вариант (обязателен в репо): каталог `wheels/` собирается командой `pip download -r requirements.txt -d wheels/` на машине с интернетом; на сервере `pip install --no-index --find-links wheels/ -r requirements.txt`. Все шрифты/JS-вендоры уже в /static.
3. Бэкап: SQLite-файл + .env, скрипт `deploy/backup.sh` (sqlite3 .backup + tar с датой).

Логи — в stdout (journald), уровень из конфига.

## 13. Требования ИБ (согласовано с «Положением о порядке использования … srv-ai»)

Базовые:
- Никакого исходящего трафика из приложения, кроме LLM_BASE_URL и RAG_BASE_URL (соответствие пп. 1.4, 3.2, 4.1 Положения: данные не покидают изолированный контур).
- Экранирование HTML при рендере markdown (защита от XSS в заметках и ответах LLM).
- CSRF: SameSite=Lax + проверка заголовка Origin на мутирующих запросах.
- Параметризованные SQL-запросы, никакой конкатенации.
- В журналы приложения (stdout) и audit_log содержание запросов/ответов и пароли не пишутся; аудит фиксирует только факты действий (кто, что, когда) — в соответствии с п. 3.7 Положения. История чатов хранится в БД как пользовательская функция и доступна только владельцу.
- Заголовки: X-Content-Type-Options, X-Frame-Options=DENY, CSP `default-src 'self'`.

Закрытие пробела п. 3.6 Положения (отсутствие аутентификации):
- Новый UI вводит обязательную аутентификацию (см. §3). **Обязательное сопутствующее требование к развёртыванию:** после ввода UI в эксплуатацию llama.cpp (:8000) и сервис эмбеддингов (:8001) должны быть привязаны к 127.0.0.1 либо закрыты правилами firewall (ufw/iptables) от всех адресов, кроме localhost и хоста LightRAG. Прямой доступ пользователей к API моделей в обход UI должен стать технически невозможным — иначе аутентификация UI не имеет смысла. Инструкция для сисадмина — в deploy/README-deploy.md.

Фильтр персональных данных (реализация п. 4.5 Положения):
- Двухуровневый модуль предобработки, без дополнительного LLM-инференса:
  1. **Регулярные выражения** для структурированных ПДн: телефоны, e-mail, СНИЛС, ИНН, серия/номер паспорта, номера банковских карт (с проверкой по Луну) и счетов. Замена на типизированные плейсхолдеры: `[ТЕЛЕФОН]`, `[EMAIL]`, `[СНИЛС]` и т.д.
  2. **NER для ФИО**: библиотека Natasha/Slovnet (лёгкая CPU-модель ~30 МБ, pip, офлайн-совместима) — извлечение сущностей PER с учётом склонений, замена на `[ФИО]`. Модель Slovnet кладётся в репозиторий/каталог данных, загрузки из сети нет.
- Точка применения: маскирование выполняется **до** отправки в LLM и **до** записи сообщения в историю чатов (в БД попадает уже очищенный текст). Инструментам LLM (заметки/календарь) передаётся тот же очищенный текст.
- Белый список в конфиге (`PII_WHITELIST_FILE`): фамилии/термины, не подлежащие маскированию (подписанты нормативных документов, устоявшиеся названия методик), чтобы снизить ложные срабатывания NER на нормативке.
- UI: если запрос был изменён фильтром — компактная плашка «Заменено N элементов ПДн» над сообщением (без раскрытия исходных значений).
- Включение флагом `PII_FILTER=true`; в метриках — счётчики срабатываний по типам, без записи содержимого.
- В список допустимых зависимостей (§2) добавляются `natasha`, `slovnet`, `navec`.

Информирование пользователя (пп. 4.3, 5.1 Положения):
- На странице входа — краткое напоминание: запрещён ввод сведений, составляющих гостайну; ответы Системы носят рекомендательный характер и подлежат проверке специалистом.
- Постоянная строка-дисклеймер под полем ввода чата: «Ответ ИИ носит рекомендательный характер и требует проверки специалистом».

Метрики (п. 3.7 Положения):
- Эндпойнт /api/admin/metrics: число обработанных запросов, успешных/неуспешных, средняя скорость генерации (токены/с из ответа llama.cpp) — без содержания запросов.

После внедрения актуализировать Положение: пп. 3.3 (собственный веб-интерфейс), 3.6 (введена аутентификация), 3.7 (введён аудит действий и хранение истории чатов), 4.5 (фильтр ПДн реализован).

## 15. Эргономика и прикладные функции

- **Библиотека специализаций:** при создании чата — выбор режима (Общий / Механообработка / Сварка / Литьё / …), подставляющего соответствующий системный промпт. Промпты хранятся в БД, редактируются админом через UI, заказчик наполняет своими наработками.
- **Обратная связь:** кнопки 👍/👎 у каждого ответа модели, при 👎 — необязательное поле «что не так». Хранится (chat_id, message_id, оценка, комментарий, специализация). Выгрузка админом в JSONL — задел под датасет для LoRA-дообучения.
- **Очередь запросов:** серверная очередь к LLM (модель обрабатывает запросы последовательно); в UI честный статус «В очереди: N» до начала стриминга. Таймаут ожидания из конфига.
- **Таблицы → Excel:** у каждой markdown-таблицы в ответе кнопка «Копировать для Excel» (TSV в буфер обмена).
- **Пустой чат:** 3–4 кликабельных примера запросов под задачи отдела (тексты примеров — в конфиге/БД, редактируются админом).
- **Масштаб:** переключатель размера шрифта (3 ступени) в профиле пользователя, хранится в настройках.

## 16. Загрузка и парсинг документов в чат

- Кнопка «Прикрепить файл» в чате. Допустимые форматы (белый список): `.txt`, `.md`, `.docx`, `.xlsx`, `.csv`, `.pdf`, `.png`, `.jpg`. Лимит размера из конфига (по умолчанию 15 МБ).
- Парсинг на бэкенде:
  - `.txt`, `.md`, `.csv` — как есть (csv → markdown-таблица);
  - `.docx` — извлечение текста и таблиц (python-docx);
  - `.xlsx` — листы → markdown-таблицы (openpyxl), с ограничением строк;
  - `.png`, `.jpg` — передаются модели напрямую как изображение (llama-server запущен с mmproj, OpenAI-совместимый формат `image_url` с base64);
  - `.pdf` — сначала попытка извлечь текстовый слой (pypdf) — это приоритетный путь, на порядки дешевле по токенам; если текстовый слой отсутствует/пуст (скан) — растеризация страниц (pdftoppm) в изображения и передача модели через vision. Лимит страниц для vision-пути из конфига (`VISION_MAX_PAGES`, по умолчанию 10) — сверх лимита предупреждение и обрезка, т.к. каждая страница стоит сотни-тысячи токенов контекста и время ViT-кодирования на CPU.
- Флаг `VISION_ENABLED=true` в конфиге; при false — fallback сканов через Tesseract OCR (tesseract-ocr-rus), чтобы система работала и на сборке llama.cpp без mmproj.
- Извлечённый **текст** проходит PII-фильтр (§13) наравне с текстом пользователя. **Изображения PII-фильтром не обрабатываются** — ПДн на сканах попадают в модель без маскирования; ограничение зафиксировать в актуализируемом Положении (п. 4.5) как известное исключение для vision-пути.
- Исходные файлы после обработки не хранятся; в БД остаётся извлечённый очищенный текст; изображения для vision-пути хранятся только на время генерации во временном каталоге.
- Контроль контекста: оценка длины (текст ~3 символа/токен + бюджет на изображения); при переполнении — предупреждение и усечение с пометкой «документ обрезан».
- ИБ: проверка расширения и MIME, парсинг во временном каталоге, никакого исполнения содержимого; docx — защита от zip-бомб (лимит распакованного размера); изображения — перекодирование через Pillow перед отправкой (отсечение вредоносных метаданных/полиглотов).
- Зависимости, добавляемые к §2: `python-docx`, `openpyxl`, `pypdf`, `Pillow`; системные: `poppler-utils`, опционально `tesseract-ocr`+`tesseract-ocr-rus` для fallback.

## 17. Порядок работы и приёмка

Этапы (каждый — рабочее состояние):
1. Каркас: auth + сессии + админка пользователей + пустой UI с навигацией в стиле ОСК.
2. Чат со стримингом и историей.
3. Заметки + календарь (UI + REST).
4. Tool calling (агентный цикл, плашки действий, подтверждение деструктивных).
5. RAG-переключатель + блок источников.
6. Эргономика (§15): специализации, обратная связь, очередь, таблицы→Excel, примеры, масштаб.
7. Загрузка и парсинг документов (§16), включая OCR-путь для сканов.
8. Аудит-лог, PII-фильтр, офлайн-деплой, тесты, README.

Критерии приёмки:
- Пользователь входит по логину/паролю; чужие личные заметки/события недоступны ни через UI, ни через API, ни через инструменты LLM.
- Запрос «создай на завтра на 10:00 совещание по нормированию и заметку с повесткой» приводит к появлению события и заметки без ручных действий.
- «Что у меня на этой неделе?» — модель отвечает по реальным данным календаря.
- При выключенном интернете на сервере всё работает (после офлайн-установки).
- Стриминг ответа начинается < 2 с после первого токена модели; UI не блокируется.
